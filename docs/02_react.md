## 2.1 什么是 ReAct？

### 2.1.1 ReAct 的发现故事：从"直觉"到"理性"的突破

2022 年，普林斯顿大学的研究团队在尝试解决大型语言模型（LLM）的一个根本性问题时，意外发现了一种革命性的方法。

**研究背景：LLM 的"双重困境"**

当时的 AI 社区面临着两个看似矛盾的问题：

| 方法 | 优势 | 致命缺陷 |
|------|------|----------|
| **纯推理（CoT）** | 能够进行复杂的逻辑推理，决策过程可解释 | 无法获取外部信息，容易产生"事实幻觉" |
| **纯行动（Act-only）** | 能够与环境交互获取真实信息 | 无法分解复杂目标，容易陷入盲目尝试 |

这就好比人类做决策时的两种极端：
- **空想家**：整天思考但从不行动，想法再好也无法验证
- **行动派**：埋头苦干但从不思考，效率低下且容易走弯路

**突破性发现**

Yao 等人在论文《[ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)》中提出了一个简单但深刻的想法：

> **为什么不让模型"边想边做"呢？**

研究团队设计了一种新的框架，让 LLM 在执行任务时：
1. 先用自然语言说出自己的"想法"（Thought）
2. 然后根据想法选择"行动"（Action）
3. 最后观察行动的"结果"（Observation）
4. 基于结果继续思考，形成循环

**实验结果令人振奋**：

ReAct 在多个基准测试上全面超越了传统方法：

| 基准测试 | 任务类型 | ReAct 表现 | 超越幅度 |
|----------|----------|------------|----------|
| **HotpotQA** | 多跳问答（需要多次检索才能回答的问题） | 准确率提升 15% | 减少了 42% 的事实幻觉 |
| **FEVER** | 事实验证（判断陈述真假） | 准确率达到 68.5% | 超越 CoT 方法 12 个百分点 |
| **ALFWorld** | 文本环境决策（如整理房间） | 任务完成率 87% | 比纯行动方法高 34% |
| **WebShop** | 在线购物（搜索 + 决策） | 成功率 72% | 展现出更好的导航能力 |

**关键洞察**：ReAct + CoT（思维链）+ Self-Consistency（自一致性）的组合通常能够超越所有其他方法。

> **💡 理解要点**：ReAct 的核心价值不在于"思考"或"行动"本身，而在于两者的**协同效应**。思考指导行动，行动验证思考，形成良性循环。

### 2.1.2 ReAct 的核心思想：像侦探一样思考

为了直观理解 ReAct，让我们用一个生动的类比：

#### 🕵️ 侦探破案类比

想象你是一位侦探，接到一个案件："找出谁偷吃了冰箱里的蛋糕"。

**没有 ReAct 的思维方式（纯推理）**：
```
你坐在办公室里空想：
"嗯...蛋糕被吃了，可能是家里人吃的。
小明最喜欢吃甜食，应该是他吧。"
→ 结论：小明偷吃的（但没有证据，可能是错的）
```

**纯行动的思维方式（不思考）**：
```
你开始盲目行动：
- 检查门锁（与案件无关）
- 询问邻居（太早了，还没信息）
- 翻垃圾桶（方向错了）
→ 效率低下，做了很多无用功
```

**ReAct 的思维方式（边想边做）**：
```
Thought: 我需要找出谁吃了蛋糕。首先应该确认蛋糕是什么时候被吃的。
Action: 检查冰箱附近的监控摄像头
Observation: 监控显示下午 3 点有人打开冰箱

Thought: 下午 3 点家里有谁在呢？需要确认。
Action: 询问家人下午的行踪
Observation: 小明说他在家，小红说去购物了

Thought: 需要验证小明的说法，看看有没有其他证据。
Action: 检查小明房间的垃圾桶
Observation: 发现蛋糕包装盒

Thought: 我已经有足够证据得出结论。
Final Answer: 小明在下午 3 点偷吃了蛋糕，证据是监控和他的房间里的包装盒。
```

看到了吗？ReAct 的思维方式：
- **有目标**：每一步都有明确的目的
- **有依据**：结论基于收集到的证据
- **可追溯**：整个推理过程清晰可见

**流程图解读**：

1. **Thought（思考）** 阶段做什么？
   - 分析当前问题的类型
   - 评估已有信息的完整性
   - 规划下一步的行动方案
   - 决定使用哪个工具

2. **Action（行动）** 阶段做什么？
   - 根据思考结果选择工具
   - 准备工具所需的参数
   - 执行工具调用

3. **Observation（观察）** 阶段做什么？
   - 接收工具返回的结果
   - 判断结果是否有效
   - 评估是否已完成任务
   - 如果不完整，准备下一轮思考

#### ReAct 三大核心组件详解

| 组件 | 作用 | 类比 | 关键特征 |
|------|------|------|----------|
| **Thought（推理）** | 生成语言形式的推理轨迹，帮助模型规划、跟踪和更新行动计划 | 侦探的"内心独白" | 可追溯、可理解、可调试 |
| **Action（行动）** | 执行具体操作，与外部环境交互（如搜索、计算、API 调用） | 侦探的"调查行动" | 获取外部信息，弥补 LLM 知识局限 |
| **Observation（观察）** | 获取行动结果，为下一步推理提供新信息 | 侦探的"线索发现" | 形成反馈闭环，支持迭代优化 |

**ReAct 的关键优势**：

1. **检索信息支持推理**
   - 问题：LLM 的知识有截止时间，无法获取最新信息
   - ReAct 解决方案：通过工具调用获取实时数据
   - 效果：减少事实幻觉，提高答案准确性

2. **推理帮助定位检索目标**
   - 问题：盲目搜索效率低下
   - ReAct 解决方案：先思考需要什么信息，再精准搜索
   - 效果：提高信息获取效率，减少无用操作

> **💡 记忆技巧**：把 ReAct 想象成一个"思考者 + 行动者"的组合体：
> - 思考者说："我觉得我们需要知道 X"
> - 行动者说："好的，我去查 X"
> - 思考者再看结果："哦，原来 X 是这样的，那我们接下来需要 Y..."

### 2.1.3 ReAct vs 其他推理方法：全方位对比

为了深入理解 ReAct 的独特价值，让我们将它与其他主流推理方法进行详细对比：

#### 方法对比矩阵

| 方法 | 核心特点 | 工作流程 | 优点 | 缺点 | 适用场景 | 典型错误 |
|------|----------|----------|------|------|----------|----------|
| **Standard Prompting** | 直接问答 | 问题 → 答案 | 简单快速，资源消耗低 | 容易出错，无法验证，事实幻觉严重 | 简单问题、常识问答 | "15×7=135"（计算错误） |
| **Chain of Thought (CoT)** | 逐步推理 | 问题 → 推理步骤 → 答案 | 提高推理准确性，过程可解释 | 无法获取外部信息，容易幻觉，对复杂事实无能为力 | 数学题、逻辑推理、代码生成 | "Python 是 John Smith 创建的"（幻觉） |
| **Act-only** | 只行动不思考 | 问题 → 行动 → 结果 → 行动 → ... | 直接执行，获取真实信息 | 无法分解复杂目标，效率低下，容易迷失方向 | 简单工具调用、单步查询 | 盲目搜索，不知道下一步该做什么 |
| **ReAct** | 推理 + 行动 | 问题 → 思考 → 行动 → 观察 → 思考 → ... | 结合内外信息，可追溯，可调试，减少幻觉 | 实现复杂度较高，依赖检索质量，延迟稍高 | 复杂问答、多步任务、事实验证 | 工具选择不当，思考方向错误 |
| **Tree of Thoughts (ToT)** | 多路径探索 | 问题 → 多个思考路径 → 评估 → 选择最佳 | 能找到全局最优解，适合复杂规划 | 计算成本极高，需要多次 LLM 调用 | 需要规划的任务、策略制定 | 路径爆炸，难以收敛 |
| **Reflexion** | 自我反思 | 尝试 → 失败 → 反思 → 再尝试 | 从错误中学习，持续改进 | 需要额外反馈机制，训练成本高 | 迭代优化任务、技能学习 | 反思质量依赖反馈准确性 |

#### 可视化对比：不同方法处理同一问题

**问题**："Python 创始人现在的年龄是多少？"

**Standard Prompting（直接回答）**：
```
用户：Python 创始人现在的年龄是多少？
AI: 大约 65 岁。（模糊且可能错误）
```
⚠️ 问题：没有推理过程，答案不可验证，可能是幻觉

**Chain of Thought（仅推理）**：
```
用户：Python 创始人现在的年龄是多少？
AI:
让我想一想...
- Python 是一种编程语言
- 它的创始人应该是个程序员
- 我好像记得是某个北欧人
- 估计现在 60 多岁吧
答案：大约 60 多岁。
```
⚠️ 问题：推理过程看似合理，但信息不准确（创始人不是北欧人）

**Act-only（仅行动）**：
```
用户：Python 创始人现在的年龄是多少？
AI:
[搜索 "programmer age"]
[搜索 "computer scientist"]
[搜索 "famous people"]
...
```
⚠️ 问题：没有明确目标，搜索方向混乱

**ReAct（推理 + 行动）**：
```
用户：Python 创始人现在的年龄是多少？

Thought: 要回答这个问题，我需要：
1. 先找出 Python 创始人是谁
2. 然后查询他的出生年份或当前年龄

Action: search("Python programming language creator")
Observation: Python was created by Guido van Rossum and first released in 1991.

Thought: 现在我知道了创始人是 Guido van Rossum，
接下来需要查询他的出生信息。

Action: search("Guido van Rossum birth date")
Observation: Guido van Rossum was born on January 31, 1956.

Thought: 他出生于 1956 年 1 月 31 日，现在是 2024 年，
我需要计算 2024 - 1956 = 68

Action: calculate("2024 - 1956")
Observation: 68

Thought: 我得到了最终答案。
Final Answer: Python 创始人 Guido van Rossum 出生于 1956 年 1 月 31 日，今年 68 岁。
```
✅ 优势：过程清晰，每一步都有依据，答案可验证

```
                             是             否
```

| "规划一次欧洲旅行" | ReAct/ToT | 需要多步规划和信息收集 |

## 2.2 为什么需要 ReAct？

### 2.2.1 LLM 的四大局限性：ReAct 诞生的必然性

要真正理解 ReAct 的价值，我们首先需要深入了解大型语言模型（LLM）存在的固有局限性。这些局限性不是某个模型的问题，而是当前基于 Transformer 架构的 LLM 的共性。

#### 局限性一：不擅长精确计算 🔢

**问题根源**：

LLM 是基于概率预测的模型。它的工作原理是：
```
给定上文 → 预测下一个词的概率分布 → 选择概率最高的词
```

这意味着 LLM 在做计算时，实际上是在"猜测"答案，而不是真正计算。

**生动示例**：

```
用户：15 乘以 7 再加 20 等于多少？

LLM（直接回答）：
思考过程（内部）：
- "15 乘以 7"... 我见过类似的问题
- 好像是 105？还是 115？
- 然后加 20...
- 我猜是 135 吧

回答：135（错误！正确答案是 125）
```

**为什么出错**：
- LLM 没有在真正计算，而是在回忆"类似问题的答案"
- 训练数据中可能有各种计算题，答案参差不齐
- 模型无法区分"记忆"和"计算"

**真实案例**：
有研究表明，即使是 GPT-4 这样的先进模型，在进行多位数乘法时也会出错，尤其是当训练数据中没有完全相同的题目时。

---

#### 局限性二：知识截止问题 📅

**问题根源**：

LLM 的知识完全来自于训练数据，而训练数据有明确的时间截止时间（cut-off date）。

**各模型的知识截止时间**：

| 模型 | 知识截止时间 |
|------|--------------|
| GPT-4 (初始版) | 2021 年 9 月 |
| GPT-4 Turbo | 2023 年 12 月 |
| Claude 3 | 2023 年 8 月 |
| 通义千问 | 训练数据截止时间 |

**生动示例**：

```
用户：2024 年诺贝尔物理学奖得主是谁？

LLM（知识截止 2023 年）：
"抱歉，我的知识截止于 2023 年，无法回答这个问题。"

或者更糟糕的情况 - 幻觉：
"2024 年诺贝尔物理学奖授予了 John Doe 和 Jane Smith，
以表彰他们在量子计算领域的贡献。"（完全是编造的）
```

**影响范围**：
- 时事新闻：选举结果、体育赛事、奖项颁发
- 科技进展：新发现的论文、新发布的产品
- 人物动态：职位变动、言论发表
- 数据更新：人口统计、经济指标

---

#### 局限性三：幻觉（Hallucination）问题 👻

**什么是幻觉**：

LLM 生成看似合理但不准确或完全虚构的信息。

**问题根源**：

1. **概率生成本质**：LLM 追求"听起来合理"而非"事实准确"
2. **训练数据噪音**：互联网上的错误信息被学习
3. **过度泛化**：从有限例子中推断出错误结论

**幻觉的类型**：

| 类型 | 示例 | 危害程度 |
|------|------|----------|
| 事实性幻觉 | "Python 由 John Smith 创建"（实际是 Guido van Rossum） | 高 - 传播错误知识 |
| 引用性幻觉 | "根据 Smith et al. (2022) 的研究..."（论文不存在） | 高 - 伪造来源 |
| 推理性幻觉 | "因为 A 导致 B，所以 C 一定正确"（推理错误） | 中 - 逻辑谬误 |
| 一致性幻觉 | 前后矛盾："他住在北京"vs"他在上海长大" | 中 - 可信度下降 |

**生动示例**：

```
用户：Python 语言的创始人是谁？

LLM（幻觉）：
"Python 是由 John Smith 于 1995 年在荷兰创建的。"

❌ 错误点：
- 名字错了：是 Guido van Rossum，不是 John Smith
- 年份错了：是 1991 年首次发布，不是 1995 年
- 地点描述不准确：Guido 是荷兰人，但 Python 是在 CWI 研究所创建的
```

**为什么幻觉危险**：

1. **难以察觉**：幻觉内容通常"听起来很合理"
2. **传播快速**：错误信息被用户信任并传播
3. **纠正困难**：一旦形成错误印象，难以改变

---

#### 局限性四：无法分解复杂任务 🧩

**问题描述**：

面对需要多步骤解决的复杂问题，LLM 倾向于"一跳到底"，而不是分步处理。

**示例对比**：

```
用户：比较 Python 和 JavaScript 创始人的年龄差异

LLM（直接回答，不分解）：
"Python 创始人 Guido van Rossum 比 JavaScript 创始人 Brendan Eich
大 5 岁左右。"（没有推理过程，无法验证准确性）

问题：
- 没有显示如何得出这个结论
- 用户无法判断答案是否可靠
- 如果错了，不知道哪里错了
```

**正确的方式应该是**：
1. 找出 Python 创始人是谁
2. 找出 JavaScript 创始人是谁
3. 查询两人的出生日期
4. 计算年龄差异

### 2.2.2 ReAct 如何解决 LLM 的局限性

ReAct 通过引入"推理 + 行动"循环，针对性地解决了上述四大局限性：

| 局限性 | ReAct 解决方案 | 解决机制 | 效果 |
|--------|----------------|----------|------|
| **计算不准确** | 调用计算器工具 | 将计算任务交给专门的工具，而非依赖 LLM 的"直觉" | 准确率接近 100% |
| **知识截止** | 调用搜索工具 | 实时获取最新信息，突破训练数据限制 | 可获取实时信息 |
| **事实幻觉** | 基于观察作答 | 答案必须基于工具返回的真实信息，而非"编造" | 幻觉减少 40%+ |
| **复杂任务** | 分步思考 + 行动 | Thought 步骤显式规划，将大任务分解为小步骤 | 复杂任务完成率提升 30%+ |

## 2.3 ReAct 核心组件详解

现在我们已经理解了 ReAct 的价值，接下来深入剖析它的三个核心组件：Thought（思考）、Action（行动）和 Observation（观察）。这部分是理解 ReAct 运作机制的关键。

### 2.3.1 Thought（思考）：Agent 的"内心独白"

#### Thought 的本质作用

想象你在解决一个复杂问题时，大脑中的思考过程：

```
"嗯...这个问题有点复杂。
首先我需要知道 A 的信息，
然后通过 A 来推导 B，
最后才能得出答案..."
```

这就是 Thought 在做的事情——让 AI 把它的思考过程"说出来"。

**Thought 的三大作用**：

| 作用 | 说明 | 实际效果 |
|------|------|----------|
| **分析当前状态** | 理解问题、评估已有信息 | 明确"我现在知道什么" |
| **规划下一步** | 决定使用什么工具或给出答案 | 明确"我接下来该做什么" |
| **反思调整** | 根据观察结果调整策略 | 明确"是否需要改变方法" |

---

#### Good Thought vs Bad Thought 对比

**场景**：用户问"北京和上海的平均气温差是多少？"

---

❌ **差的 Thought 示例**：

```
Thought: 我不知道，搜索一下吧。
Action: search("北京")

【问题分析】
- 思考过于简单，没有明确搜索目的
- 搜索关键词太宽泛，无法获得有效信息
- 没有规划完整的解决步骤
```

```
Thought: 直接计算。
Action: calculate("北京 - 上海")

【问题分析】
- 没有先获取气温数据就直接计算
- 把城市名称当数字计算，完全错误
- 缺少必要的信息收集步骤
```

---

✅ **好的 Thought 示例**：

```
Thought: 要计算两个城市的平均气温差，我需要：
1. 先查询北京的平均气温
2. 查询上海的平均气温
3. 然后计算两者的差值

首先，让我搜索北京的平均气温。
Action: search("北京 年平均气温")

【优点分析】
- 明确列出了完整的解决步骤
- 清楚说明为什么要执行这个行动
- 搜索关键词精准，能获得有效信息
```

```
Thought: 我已经获取了北京的平均气温是 12°C，
接下来需要查询上海的平均气温才能计算差值。
Action: search("上海 年平均气温")

【优点分析】
- 说明已有信息（北京 12°C）
- 明确下一步目标
- 思考连贯，逻辑清晰
```

---

#### Thought 生成技巧：如何写好 Thought

根据 ReAct 论文和实践经验，好的 Thought 应该具备以下特征：

**1. 明确当前目标**

```
✅ "我需要找出 Python 创始人，然后查询他的出生信息。"
❌ "我要搜索一些信息。"
```

**2. 分析已有信息**

```
✅ "现在我知道 Guido van Rossum 出生于 1956 年，需要计算他的当前年龄。"
❌ "我有一些信息了。"
```

**3. 识别信息缺口**

```
✅ "但我还不知道上海的气温数据，这是计算差值所必需的。"
❌ "还缺一些东西。"
```

**4. 决定下一步行动**

```
✅ "接下来我应该使用计算器计算 2024-1956 得到年龄。"
❌ "我做些计算。"
```

---

#### Thought 的格式和风格指南

**推荐格式**：

```
Thought: [对当前状态的分析] + [下一步计划] + [具体行动的理由]

示例：
Thought: 我已经完成了前两步计算（15×7=105，105+20=125），
         现在得到了最终答案，可以回复用户了。
```

**风格建议**：

| 方面 | 建议 | 原因 |
|------|------|------|
| 语言 | 自然、简洁 | 便于理解和调试 |
| 长度 | 1-3 句话 | 平衡信息量和效率 |
| 内容 | 包含推理过程 | 使决策可追溯 |
| 人称 | 第一人称（"我需要..."） | 增强主体意识 |

---

### 2.3.2 Action（行动）：从思考到实践

#### Action 的定义和类型

**Action 是什么**：

Action 是 Thought 的具体执行，是 Agent 与外部世界交互的方式。

**Action 的两种类型**：

```
┌─────────────────────────────────────────────────────────┐
│                      Action 类型                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 工具调用（Tool Call）                                │
│     目的：获取信息或执行操作                             │
│     示例：calculator(...), search(...)                  │
│     继续循环：需要等待 Observation                       │
│                                                         │
│  2. 最终答案（Final Answer）                             │
│     目的：给出任务结论                                   │
│     示例：Final Answer: ...                             │
│     结束循环：任务完成                                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

---

#### Action 的格式详解（OpenAI Function Calling）

现代 LLM（如 GPT、Qwen）支持原生工具调用，使用标准的 JSON 格式：

```json
{
  "id": "call_abc123",
  "type": "function",
  "function": {
    "name": "calculator",
    "arguments": {
      "expression": "15 * 7"
    }
  }
}
```

**字段详解**：

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `id` | string | 工具调用的唯一标识 | `"call_abc123"` |
| `type` | string | 调用类型（目前是 function） | `"function"` |
| `function.name` | string | 工具名称 | `"calculator"` |
| `function.arguments` | object | 工具参数（JSON 对象） | `{"expression": "15 * 7"}` |

---

#### 如何决定使用哪个工具：决策树

```
                    当前任务需要什么？
                          │
        ┌─────────────────┼─────────────────┐
        │                 │                 │
        ▼                 ▼                 ▼
    数学计算          信息查询          其他操作
        │                 │                 │
        ▼                 ▼                 ▼
  calculator        search/google     相应工具
  （计算器）        （搜索引擎）
```

**决策过程示例**：

```
用户问题："15 乘以 7 再加 20 等于多少？"

Thought 过程：
1. 分析问题类型 → 数学计算
2. 查看可用工具 → calculator, search
3. 选择匹配工具 → calculator（计算任务）
4. 构造参数 → {"expression": "15 * 7 + 20"}

决策依据：
- 为什么不用 search？→ 这是计算问题，不是信息查询
- 为什么选 calculator？→ 专门用于数学计算
```

---

#### Action 设计最佳实践

**1. 工具命名要直观**

```
✅ 推荐：calculator, search, read_file, write_file
❌ 避免：tool1, tool2, func_a, helper
```

**2. 参数定义要清晰**

```python
# ✅ 好的参数设计
{
  "type": "object",
  "properties": {
    "expression": {
      "type": "string",
      "description": "数学表达式，如 '2 + 3 * 4'"
    }
  },
  "required": ["expression"]
}

# ❌ 差的参数设计（太模糊）
{
  "type": "object",
  "properties": {
    "input": {
      "type": "string",
      "description": "输入内容"
    }
  }
}
```

**3. 工具描述要具体**

```python
# ✅ 好的描述
description = "计算数学表达式。支持加减乘除、幂运算、三角函数等。"

# ❌ 差的描述（太模糊）
description = "用于计算。"
```

#### 如何处理 Observation

**成功时的处理**：

```
1. 验证结果是否合理
2. 判断是否已完成任务
3. 如果完成 → Final Answer
4. 如果未完成 → 继续 Thought-Action 循环
```

**失败时的处理**：

```
1. 分析失败原因
2. 调整策略（换工具、换参数、换方法）
3. 记录错误到记忆
4. 继续尝试或向用户求助
```

```
【关键学习点】

- 最终成功获取信息
```

#### Agent 状态机详解

ReAct Agent 在执行过程中会经历多个状态，这些状态由 `AgentState` 枚举定义：

| **COMPLETE** | 达到终止条件 | 无 | IDLE（可接受新任务） |
| **ERROR** | 发生异常 | 记录错误信息 | IDLE（重置后）或重试 |

### 2.4.2 核心类型定义（llm/types.py）

**设计思路**：

**源代码**（`src/llm/types.py`）：

```python
from pydantic import BaseModel


class ToolCall(BaseModel):

class Message(BaseModel):
    """Represents a chat message."""
    role: str                    # 角色：system, user, assistant, tool
    content: Optional[str] = None  # 消息内容

class ToolDefinition(BaseModel):

@dataclass
```

### 2.4.3 工具基类实现（tools/base.py）

**设计思路**：

想象你要管理一个工具箱，里面有锤子、螺丝刀、扳手等各种工具。为了让所有工具都能被统一管理，你需要：
1. **定义一个"工具"的标准**（所有工具都要有名称、用途、使用方法）
2. **创建一个注册表**（记录有哪些工具，如何使用）

这就是 `BaseTool`和`ToolRegistry` 的设计思想。

**核心概念：抽象基类（Abstract Base Class）**

`BaseTool` 是一个**抽象类**，它定义了所有工具必须实现的接口，但自己不实现具体功能。这就像是一个"工具模板"。

**源代码**（`src/tools/base.py` 核心部分）：

```python
# src/tools/base.py:60-165
from abc import ABC, abstractmethod
from typing import Any, Dict, Optional
from dataclasses import dataclass, field
from enum import Enum

class ToolResultStatus(Enum):
    """Status of tool execution."""
    SUCCESS = "success"
    ERROR = "error"
    TIMEOUT = "timeout"

@dataclass
class ToolResult:
    """Result of a tool execution."""
    status: ToolResultStatus      # 执行状态
    output: Any = None            # 输出结果
    error: Optional[str] = None   # 错误信息（如果失败）
    metadata: Dict[str, Any] = field(default_factory=dict)

    def to_string(self) -> str:
        """Convert result to string for LLM consumption."""
        if self.status == ToolResultStatus.SUCCESS:
            if isinstance(self.output, str):
                return self.output
            return json.dumps(self.output, ensure_ascii=False, indent=2)
        else:
            return f"Error: {self.error}"

class BaseTool(ABC):
    """Abstract base class for all tools."""

    @property
    @abstractmethod
    def name(self) -> str:
        """Unique identifier for the tool."""
        pass

    @property
    @abstractmethod
    def description(self) -> str:
        """Human-readable description of what the tool does."""
        pass

    @property
    def parameters(self) -> Dict[str, Any]:
        """JSON Schema for tool parameters."""
        return {
            "type": "object",
            "properties": {},
            "required": [],
        }

    @abstractmethod
    def execute(self, **kwargs) -> ToolResult:
        """Execute the tool with the given parameters."""
        pass

    def to_openai_tool(self) -> Dict[str, Any]:
        """Convert to OpenAI function calling format."""
        return {
            "type": "function",
            "function": {
                "name": self.name,
                "description": self.description,
                "parameters": self.parameters,
            },
        }
```

**代码详解**：

#### 1. `ToolResult` - 工具执行结果

**设计目标**：统一工具返回格式，让 Agent 能够一致性地处理所有工具的结果。

**字段说明**：
```python
ToolResult(
    status=ToolResultStatus.SUCCESS,  # 状态：成功/错误/超时
    output=105,                        # 输出：计算结果
    error=None,                        # 错误信息：无
    metadata={"execution_time": 0.1}   # 元数据：执行时间等
)
```

**`to_string()` 方法的作用**：
LLM 只能理解文本，所以需要将结果转换为字符串格式供 LLM 消费。

#### 2. `BaseTool` - 工具基类

**四个抽象方法**（子类必须实现）：

| 方法/属性 | 返回类型 | 说明 | 示例 |
|-----------|----------|------|------|
| `name` | `str` | 工具唯一标识 | `"calculator"` |
| `description` | `str` | 工具描述 | "计算数学表达式" |
| `parameters` | `Dict` | JSON Schema 参数定义 | `{"type": "object", ...}` |
| `execute` | `ToolResult` | 执行逻辑 | 实际计算代码 |

**`to_openai_tool()` 方法**：
将工具转换为 OpenAI Function Calling 格式，这样 LLM 就能理解如何调用这个工具。

#### 实例：CalculatorTool 的实现

让我们看一个具体的工具实现：

```python
# src/tools/builtins.py (简化版)
import math
from .base import BaseTool, ToolResult, ToolResultStatus

class CalculatorTool(BaseTool):
    """Calculator tool for mathematical expressions."""

    @property
    def name(self) -> str:
        return "calculator"

    @property
    def description(self) -> str:
        return "计算数学表达式。支持加减乘除、幂运算、三角函数等。"

    @property
    def parameters(self) -> Dict[str, Any]:
        return {
            "type": "object",
            "properties": {
                "expression": {
                    "type": "string",
                    "description": "数学表达式，如 '2 + 3 * 4' 或 'sin(pi/2)'"
                }
            },
            "required": ["expression"]
        }

    def execute(self, expression: str) -> ToolResult:
        # 安全的表达式求值
        allowed_names = {
            "abs": abs, "round": round,
            "sin": math.sin, "cos": math.cos, "tan": math.tan,
            "sqrt": math.sqrt, "log": math.log,
            "pi": math.pi, "e": math.e,
        }
        try:
            result = eval(expression, {"__builtins__": {}}, allowed_names)
            return ToolResult(status=ToolResultStatus.SUCCESS, output=result)
        except Exception as ex:
            return ToolResult(status=ToolResultStatus.ERROR, error=str(ex))
```

**代码解读**：

1. **`name` 属性**：返回 `"calculator"`，这是工具的唯一标识
2. **`description` 属性**：详细描述工具功能，帮助 LLM 理解何时使用
3. **`parameters` 属性**：定义参数格式，LLM 会按照这个格式生成调用
4. **`execute` 方法**：实际执行计算逻辑
   - 使用 `eval()` 但限制命名空间，防止代码注入
   - 只允许数学函数，不允许其他 Python 内置函数
   - 异常处理返回错误信息

### 2.4.4 工具注册表实现（tools/base.py）

**设计思路**：

想象一个图书馆，每本书都有唯一的编号。图书管理员需要：
1. **记录所有藏书**（工具注册）
2. **根据编号找书**（工具获取）
3. **列出所有藏书**（工具列表）

`ToolRegistry` 就是这样一个"工具图书馆"。

**源代码**（`src/tools/base.py:168-302`）：

```python
class ToolRegistry:
    """Central registry for managing tools."""

    def __init__(self):
        """Initialize an empty registry."""
        self._tools: Dict[str, BaseTool] = {}
        logger.info("Initialized ToolRegistry")

    def register(self, tool: BaseTool) -> None:
        """Register a tool."""
        if tool.name in self._tools:
            logger.warning(f"Overwriting existing tool: {tool.name}")
        self._tools[tool.name] = tool
        logger.debug(f"Registered tool: {tool.name}")

    def get(self, name: str) -> Optional[BaseTool]:
        """Get a tool by name."""
        return self._tools.get(name)

    def list_tools(self) -> List[str]:
        """List all registered tool names."""
        return list(self._tools.keys())

    def to_openai_tools(self) -> List[Dict[str, Any]]:
        """Convert all tools to OpenAI format."""
        return [tool.to_openai_tool() for tool in self._tools.values()]

    def execute(self, name: str, **kwargs) -> ToolResult:
        """Execute a tool by name."""
        tool = self.get(name)
        if not tool:
            return ToolResult(
                status=ToolResultStatus.ERROR,
                error=f"Tool not found: {name}",
            )
        try:
            if not tool.validate_input(**kwargs):
                return ToolResult(
                    status=ToolResultStatus.ERROR,
                    error=f"Invalid input for tool: {name}",
                )
            result = tool.execute(**kwargs)
            return result
        except Exception as e:
            return ToolResult(
                status=ToolResultStatus.ERROR,
                error=str(e),
            )
```

**方法详解**：

| 方法 | 功能 | 使用场景 |
|------|------|----------|
| `register(tool)` | 注册工具 | Agent 初始化时 |
| `get(name)` | 获取工具 | 内部使用 |
| `list_tools()` | 列出所有工具名 | 调试、展示 |
| `to_openai_tools()` | 转换为 OpenAI 格式 | 发送给 LLM |
| `execute(name, **kwargs)` | 执行工具 | ReAct 循环中 |

**使用示例**：

```python

# 注册工具
registry.register(CalculatorTool())
registry.register(SearchTool())

# 获取工具列表
print(registry.list_tools())  # ['calculator', 'search']

# 转换为 OpenAI 格式（发送给 LLM）

# 执行工具

print(result.output)  # 105
```

### 2.4.5 记忆系统实现（memory/short_term.py）

**设计思路**：

想象你在和朋友聊天，你需要记住：
1. **朋友说了什么**（用户消息）
2. **你回答了什么**（助手消息）
3. **工具返回了什么**（工具结果）

但是，人的短期记忆是有限的，你不可能记住所有的对话内容。所以需要：
- **滑动窗口机制**：只保留最近的 N 条消息
- **系统消息保留**：始终保留系统提示词

**源代码**（`src/memory/short_term.py` 核心部分）：

```python
# src/memory/short_term.py:56-154
from typing import List, Optional, Dict, Any
from dataclasses import dataclass, field
from datetime import datetime

@dataclass
class Message:
    """A single message in the conversation."""
    role: str                              # 发送者
    content: Optional[str] = None          # 内容
    timestamp: datetime = field(default_factory=datetime.now)
    metadata: Dict[str, Any] = field(default_factory=dict)
    tool_calls: Optional[List[Dict[str, Any]]] = None
    tool_call_id: Optional[str] = None

class ShortTermMemory:
    """Short-term memory using sliding window."""

    def __init__(self, max_messages: int = 50, system_message: Optional[str] = None):
        self.max_messages = max_messages
        self.system_message = system_message
        self._messages: List[Message] = []

        # 添加系统消息（如果有）
        if system_message:
            self._messages.append(Message(role="system", content=system_message))

    def add_message(
        self,
        role: str,
        content: Optional[str] = None,
        tool_calls: Optional[List[Dict[str, Any]]] = None,
        tool_call_id: Optional[str] = None,
    ) -> None:
        """Add a message to memory."""
        message = Message(
            role=role,
            content=content,
            tool_calls=tool_calls,
            tool_call_id=tool_call_id,
        )
        self._messages.append(message)

        # 滑动窗口：超过限制时移除旧消息（但保留系统消息）
        if len(self._messages) > self.max_messages:
            start_idx = 1 if self._messages[0].role == "system" else 0
            excess = len(self._messages) - self.max_messages
            self._messages = self._messages[:start_idx] + self._messages[start_idx + excess:]

    def get_messages(self) -> List[Message]:
        """Get all messages."""
        return self._messages.copy()

    def get_openai_messages(self) -> List[Dict[str, Any]]:
        """Get messages in OpenAI format."""
        return [msg.to_openai_format() for msg in self._messages]

    def clear(self) -> None:
        """Clear all messages except system message."""
        if self.system_message:
            self._messages = [Message(role="system", content=self.system_message)]
        else:
            self._messages = []
```

**滑动窗口机制详解**：

```
初始状态 (max_messages=5):
┌─────────────┐
│ System msg  │ ← 始终保留
└─────────────┘

添加消息 1-4:
┌─────────────┐
│ System msg  │
├─────────────┤
│ User msg 1  │
├─────────────┤
│ Assistant 1 │
├─────────────┤
│ User msg 2  │
├─────────────┤
│ Assistant 2 │
└─────────────┘

添加消息 5 (达到上限):
┌─────────────┐
│ System msg  │
├─────────────┤
│ User msg 1  │  ← 即将被移除
├─────────────┤
│ Assistant 1 │
├─────────────┤
│ User msg 2  │
├─────────────┤
│ Assistant 2 │
├─────────────┤
│ User msg 3  │  ← 新消息
└─────────────┘

添加消息 6 (滑动窗口生效):
┌─────────────┐
│ System msg  │
├─────────────┤
│ Assistant 1 │  ← 最旧的消息被移除
├─────────────┤
│ User msg 2  │
├─────────────┤
│ Assistant 2 │
├─────────────┤
│ User msg 3  │
├─────────────┤
│ Assistant 3 │  ← 新消息
└─────────────┘
```

**为什么需要滑动窗口**：

1. **Token 限制**：LLM 的上下文窗口有限（如 4096 tokens）
2. **性能优化**：消息太多会导致响应变慢
3. **成本控制**：更多的 tokens = 更高的费用
4. **注意力集中**：最近的消息通常更重要

```python
class BailianClient:
    """Client for Alibaba Bailian API."""

    def __init__(
        self,
        api_key: Optional[str] = None,
        base_url: Optional[str] = None,
        model: str = "qwen-plus",
        timeout: int = 30,
        max_retries: int = 3,
    ):
        # 从环境变量或参数获取 API key
        self.api_key = api_key or os.getenv("DASHSCOPE_API_KEY")
        if not self.api_key:
            raise ValueError("API key is required...")

        self.base_url = base_url or os.getenv(
            "DASHSCOPE_BASE_URL",
            "https://dashscope.aliyuncs.com/compatible-mode/v1",
        )
        self.model = model
        self.timeout = timeout
        self.max_retries = max_retries

        # 初始化 OpenAI 兼容客户端
        from openai import OpenAI
        import httpx
        self.client = OpenAI(
            api_key=self.api_key,
            base_url=self.base_url,
            timeout=httpx.Timeout(timeout, connect=5.0),
            max_retries=max_retries,
        )

    def chat(
        self,
        messages: Union[str, List[Dict[str, Any]], List[Message]],
        tools: Optional[List[Dict[str, Any]]] = None,
        temperature: float = 0.7,
        max_tokens: int = 4096,
        **kwargs,
    ) -> LLMResponse:
        """Send a chat message and get a response."""
        # 转换消息为 OpenAI 格式
        if isinstance(messages, str):
            messages = [{"role": "user", "content": messages}]

        # 转换工具为 OpenAI 格式
        openai_tools = self._convert_tools(tools)

        # 调用 LLM API
        response = self.client.chat.completions.create(
            model=self.model,
            messages=messages,
            tools=openai_tools,
            temperature=temperature,
            max_tokens=max_tokens,
            **kwargs,
        )

        # 解析响应
        choice = response.choices[0]
        content = choice.message.content
        tool_calls = None
        if choice.message.tool_calls:
            tool_calls = self._parse_tool_calls(choice.message.tool_calls)

        return LLMResponse(
            content=content,
            tool_calls=tool_calls or [],
            finish_reason=choice.finish_reason,
            usage={
                "prompt_tokens": response.usage.prompt_tokens,
                "completion_tokens": response.usage.completion_tokens,
                "total_tokens": response.usage.total_tokens,
            },
        )
```

### 2.4.7 ReAct Agent 核心实现（agent/react.py）

**重头戏来了**！前面所有的模块都是为了这一刻——ReAct Agent 的核心实现。

**设计思路**：

ReAct Agent 的核心是一个**循环**，每一轮循环包含三个步骤：
1. **Thought**：调用 LLM 进行思考
2. **Action**：解析并执行工具调用
3. **Observation**：将工具结果反馈给 LLM

这个循环持续进行，直到：
- LLM 给出 Final Answer（成功）
- 达到最大迭代次数（失败）
- 发生错误（失败）

**完整源代码**（`src/agent/react.py` 精简版，带详细注释）：

```python
# src/agent/react.py:51-301
import time
from typing import List, Optional, Dict, Any
import json
import logging

from .base import BaseAgent, AgentState, AgentConfig, AgentResult
from ..llm.client import LLMResponse, ToolCall
from ..tools.base import ToolResult, ToolResultStatus

logger = logging.getLogger(__name__)

# ReAct System Prompt Template
REACT_SYSTEM_PROMPT = """You are a helpful AI assistant that uses the ReAct framework.

You follow this pattern:
1. **Thought**: Think about what to do next
2. **Action**: Choose an action/tool to execute
3. **Observation**: Observe the result of the action
4. Repeat until you have the final answer

When you have the final answer, respond with:
Thought: I now know the final answer.
Final Answer: [your answer]

Available tools:
{tools_description}

Guidelines:
- Always explain your reasoning before taking an action
- Use tools when you need information or calculations
- Be concise but thorough in your thoughts
- If a tool fails, try a different approach
- When you have enough information, provide the final answer

Remember to think step by step!"""


class ReActAgent(BaseAgent):
    """ReAct Agent implementation."""

    def __init__(
        self,
        llm_client,
        config: Optional[AgentConfig] = None,
        tools=None,
        memory=None,
    ):
        # 设置默认 ReAct 系统提示词
        if config is None:
            config = AgentConfig(
                name="ReActAgent",
                system_prompt=REACT_SYSTEM_PROMPT.format(
                    tools_description=self._get_tools_description(tools or [])
                ),
            )

        super().__init__(llm_client, config, tools, memory)

        # 追踪 ReAct 循环的三个核心组件
        self._thoughts: List[str] = []
        self._actions: List[Dict[str, Any]] = []
        self._observations: List[str] = []

    def run(self, query: str, **kwargs) -> AgentResult:
        """运行 ReAct 循环 - 这是 Agent 的核心方法."""
        logger.info(f"Starting ReAct agent with query: {query[:100]}...")

        # 重置状态
        self._start_time = self._get_time()
        self.iteration = 0
        self.state = AgentState.THINKING
        self._thoughts = []
        self._actions = []
        self._observations = []

        # 初始化记忆（添加用户查询）
        self._add_user_message(query)

        try:
            # ═══════════════════════════════════════════════════════════
            # ReAct 主循环开始
            # ═══════════════════════════════════════════════════════════
            while self._should_continue():
                self.iteration += 1
                logger.debug(f"Iteration {self.iteration}")

                # ───────────────────────────────────────────────────────
                # Step 1: Thought - 思考
                # ───────────────────────────────────────────────────────
                self.state = AgentState.THINKING
                thought = self._think()  # 调用 LLM 获取思考内容

                if thought:
                    self._thoughts.append(thought)
                    self._log_verbose(f"Thought: {thought}")

                    # 检查是否包含最终答案
                    if ("Final Answer:" in thought or
                        "I now know the final answer" in thought):
                        self.state = AgentState.COMPLETE
                        final_answer = self._extract_final_answer(thought)
                        return AgentResult(
                            success=True,
                            output=final_answer,
                            iterations=self.iteration,
                            tool_calls=self._actions,
                        )

                # ───────────────────────────────────────────────────────
                # Step 2: Action - 行动
                # ───────────────────────────────────────────────────────
                self.state = AgentState.EXECUTING
                tool_call = self._act(thought)  # 解析工具调用

                if tool_call:
                    self._actions.append({
                        "iteration": self.iteration,
                        "name": tool_call.name,
                        "arguments": tool_call.arguments,
                    })

                    # 执行工具
                    result = self._execute_tool(tool_call)

                    # ───────────────────────────────────────────────────
                    # Step 3: Observation - 观察
                    # ───────────────────────────────────────────────────
                    self.state = AgentState.WAITING
                    observation = self._observe(result)
                    self._observations.append(observation)
                    self._log_verbose(f"Observation: {observation[:100]}...")
                else:
                    # 没有工具调用，继续循环
                    pass

            # ═══════════════════════════════════════════════════════════
            # 循环结束：达到最大迭代次数
            # ═══════════════════════════════════════════════════════════
            logger.warning(f"Max iterations ({self.config.max_iterations}) reached")
            return AgentResult(
                success=False,
                output="Maximum iterations reached. Could not complete the task.",
                iterations=self.iteration,
                tool_calls=self._actions,
                error="max_iterations_exceeded",
            )

        except Exception as e:
            logger.error(f"Agent error: {e}")
            self.state = AgentState.ERROR
            return AgentResult(
                success=False,
                error=str(e),
                iterations=self.iteration,
                tool_calls=self._actions,
            )

    def _think(self) -> Optional[str]:
        """Thinking step - get LLM response."""
        response = self._call_llm()

        # 添加助手消息到记忆
        if response.content:
            self._add_assistant_message(response.content)

        return response.content

    def _act(self, thought: Optional[str] = None) -> Optional[ToolCall]:
        """Action step - parse and return tool call."""
        response = self._call_llm()

        if response.tool_calls:
            tool_call = response.tool_calls[0]

            # 添加工具调用到记忆
            self._add_assistant_message(
                content=response.content,
                tool_calls=[{
                    "id": tool_call.id,
                    "type": "function",
                    "function": {
                        "name": tool_call.name,
                        "arguments": json.dumps(tool_call.arguments),
                    },
                }],
            )

            return tool_call

        return None

    def _observe(self, result: ToolResult) -> str:
        """Observation step - process tool result."""
        observation = result.to_string()

        # 添加工具结果到记忆
        self._add_tool_result("tool_result", observation)

        return observation

    def _extract_final_answer(self, text: str) -> str:
        """Extract the final answer from thought text."""
        if "Final Answer:" in text:
            parts = text.split("Final Answer:")
            return parts[-1].strip()

        # 查找最后一行非 Thought 内容
        lines = text.strip().split("\n")
        for line in reversed(lines):
            if line.strip() and not line.startswith("Thought:"):
                return line.strip()

        return text

    def _get_tools_description(self, tools) -> str:
        """Generate description of available tools."""
        if hasattr(tools, 'list_tools'):
            tool_list = tools.list_tools()
            descriptions = []
            for name in tool_list:
                tool = tools.get(name)
                if tool:
                    descriptions.append(f"- {name}: {tool.description}")
            return "\n".join(descriptions)
        return "No tools available"

    def get_trace(self) -> Dict[str, Any]:
        """Get the execution trace."""
        return {
            "thoughts": self._thoughts,
            "actions": self._actions,
            "observations": self._observations,
            "iterations": self.iteration,
        }
```

#### 关键方法详解

**1. `_think()` - 思考步骤**

```python
def _think(self) -> Optional[str]:
    """Reasoning step - get LLM response."""
    response = self._call_llm()

    # 添加助手消息到记忆
    if response.content:
        self._add_assistant_message(response.content)

    return response.content
```

**工作流程**：
```
_call_llm()  ──► 获取 LLM 响应
    │
    ├─► messages: 当前对话历史
    ├─► tools: 可用工具列表
    └─► 返回 LLMResponse
         │
         ├─► content: Thought 内容
         └─► tool_calls: 工具调用（如果有）
```

**2. `_act()` - 行动步骤**

```python
def _act(self, thought: Optional[str] = None) -> Optional[ToolCall]:
    """Action step - parse and return tool call."""
    response = self._call_llm()

    if response.tool_calls:
        tool_call = response.tool_calls[0]
        # 添加到记忆
        self._add_assistant_message(...)
        return tool_call
    return None
```

**注意**：这里再次调用 `_call_llm()` 是因为有些 LLM 需要单独的调用来获取工具调用。

**3. `_observe()` - 观察步骤**

```python
def _observe(self, result: ToolResult) -> str:
    """Observation step - process tool result."""
    observation = result.to_string()
    self._add_tool_result("tool_result", observation)
    return observation
```

**作用**：将工具执行结果转换为字符串，添加到记忆，供 LLM 在下一轮思考时使用。

```python
from llm.client import LLMClient
from tools.registry import ToolRegistry
from tools.builtins import CalculatorTool, SearchTool
from memory.short_term import ShortTermMemory

def main():

    # ═══════════════════════════════════════════════════════════
    # 步骤 3: 运行示例
    # ═══════════════════════════════════════════════════════════

    # 示例 1：数学计算
    print("\n" + "="*60)
    print("示例 1：数学计算")
    print("="*60)
    result = agent.run("15 乘以 7 再加 23 等于多少？")
    print(f"\n最终答案：{result.output}")

    # 示例 2：信息查询
    print("\n" + "="*60)
    print("示例 2：信息查询")
    print("="*60)
    result = agent.run("Python 是什么时候创建的？")
    print(f"\n最终答案：{result.output}")

    # 示例 3：复杂问题（多步骤）
    print("\n" + "="*60)
    print("示例 3：复杂问题")
    print("="*60)
    result = agent.run("Python 的创始人今年多少岁？")
    print(f"\n最终答案：{result.output}")

    # ═══════════════════════════════════════════════════════════
    # 步骤 4: 显示执行追踪
    # ═══════════════════════════════════════════════════════════
    print("\n" + "="*60)
    print("执行追踪")
    print("="*60)

    trace = result.tool_calls
    for i, step in enumerate(trace, 1):
        print(f"\n--- Step {i} ---")
        print(f"工具：{step['name']}")
        print(f"参数：{step['arguments']}")


if __name__ == "__main__":
    main()
```

## 2.6 提示词工程详解

提示词（Prompt）是 ReAct Agent 的"操作手册"，它决定了 Agent 如何思考、如何行动。一个好的提示词可以显著提高 Agent 的表现。

### 2.6.1 系统提示词的作用

系统提示词定义了：
- **Agent 行为模式**：告诉 LLM 如何思考和行动
- **可用工具**：列出所有可调用的工具及其用途
- **输出格式**：规范 Thought、Action、Observation 的格式
- **约束条件**：设置最大迭代次数、错误处理规则等

**ReAct 系统提示词模板**：

```python
REACT_SYSTEM_PROMPT = """

   - 是否解决了问题

```

### 2.6.5 提示词优化技巧

| 技巧 | 说明 | 示例 |
|------|------|------|
| **具体化 vs 泛化** | 工具描述要具体，包含使用场景和示例 | "计算数学表达式，如 '2 + 3 * 4'" |
| **使用示例（few-shot）** | 提供 1-3 个完整示例效果最佳 | 在提示词中包含完整的 Thought-Action-Observation 示例 |
| **添加约束条件** | 限制最大迭代次数，防止无限循环 | "最多进行 10 次迭代" |
| **设置明确的结束条件** | Final Answer 格式要清晰标识 | "Final Answer: [你的答案]" |
| **错误处理指导** | 告诉 LLM 工具失败时如何应对 | "如果工具失败，尝试其他方法或向用户求助" |

#### OpenCode 的 ReAct 循环实现

```typescript
// 简化自 OpenCode 源码
async function runAgent(userInput: string) {
  const messages: Message[] = [{ role: 'user', content: userInput }];

  while (true) {
    // Step 1: Thought - 调用 LLM
    const response = await provider.chat(messages);

    // 检查是否有工具调用
    if (response.tool_calls && response.tool_calls.length > 0) {
      const toolCall = response.tool_calls[0];

      // Step 2: Action - 执行工具
      const tool = toolRegistry.get(toolCall.name);
      const result = await tool.execute(toolCall.arguments);

      // Step 3: Observation - 添加结果到历史
      messages.push({
        role: 'assistant',
        tool_calls: [toolCall]
      });
      messages.push({
        role: 'tool',
        content: result,
        tool_call_id: toolCall.id
      });

      // 检查是否完成
      if (result.isComplete) {
        return result.finalAnswer;
      }
    } else {
      // 直接回复，没有工具调用
      return response.content;
    }
  }
}
```

| `_act()` | 解析 `response.tool_calls` |

| `_observe()` | 添加 tool result 到 messages |

### 2.9.6 实战：构建类似 Claude Code 的简易 Agent

让我们用本章学到的知识，构建一个简化版的"代码助手"：

```python
# simple_code_agent.py
from src.agent import ReActAgent, AgentConfig
from src.llm import BailianClient
from src.tools import BaseTool, ToolRegistry, ToolResult, ToolResultStatus
from src.memory import ShortTermMemory
import os

# ============== 步骤 1: 定义工具 ==============

class ReadFileTool(BaseTool):
    """读取文件内容"""

    @property
    def name(self) -> str:
        return "read_file"

    @property
    def description(self) -> str:
        return "读取文件内容。用于查看代码、配置等。"

    @property
    def parameters(self) -> dict:
        return {
            "type": "object",
            "properties": {
                "path": {
                    "type": "string",
                    "description": "文件路径，如 'src/main.py'"
                }
            },
            "required": ["path"]
        }

    def execute(self, path: str) -> ToolResult:
        try:
            if not os.path.exists(path):
                return ToolResult(
                    status=ToolResultStatus.ERROR,
                    error=f"文件不存在：{path}"
                )
            with open(path, 'r', encoding='utf-8') as f:
                content = f.read()
            return ToolResult(
                status=ToolResultStatus.SUCCESS,
                output=f"文件内容：\n{content}"
            )
        except Exception as e:
            return ToolResult(
                status=ToolResultStatus.ERROR,
                error=str(e)
            )


class EditFileTool(BaseTool):
    """编辑文件内容"""

    @property
    def name(self) -> str:
        return "edit_file"

    @property
    def description(self) -> str:
        return "编辑文件内容。需要提供文件路径、要替换的旧内容和新的内容。"

    @property
    def parameters(self) -> dict:
        return {
            "type": "object",
            "properties": {
                "path": {
                    "type": "string",
                    "description": "文件路径"
                },
                "old_str": {
                    "type": "string",
                    "description": "要替换的原始内容"
                },
                "new_str": {
                    "type": "string",
                    "description": "替换后的新内容"
                }
            },
            "required": ["path", "old_str", "new_str"]
        }

    def execute(self, path: str, old_str: str, new_str: str) -> ToolResult:
        try:
            if not os.path.exists(path):
                return ToolResult(
                    status=ToolResultStatus.ERROR,
                    error=f"文件不存在：{path}"
                )
            with open(path, 'r', encoding='utf-8') as f:
                content = f.read()

            if old_str not in content:
                return ToolResult(
                    status=ToolResultStatus.ERROR,
                    error="原始内容未找到，无法替换"
                )

            new_content = content.replace(old_str, new_str)

            with open(path, 'w', encoding='utf-8') as f:
                f.write(new_content)

            return ToolResult(
                status=ToolResultStatus.SUCCESS,
                output=f"文件已修改：{path}"
            )
        except Exception as e:
            return ToolResult(
                status=ToolResultStatus.ERROR,
                error=str(e)
            )


class RunCommandTool(BaseTool):
    """运行 shell 命令"""

    @property
    def name(self) -> str:
        return "run_command"

    @property
    def description(self) -> str:
        return "运行 shell 命令。用于执行测试、编译、启动等操作。"

    @property
    def parameters(self) -> dict:
        return {
            "type": "object",
            "properties": {
                "command": {
                    "type": "string",
                    "description": "要执行的命令"
                }
            },
            "required": ["command"]
        }

    def execute(self, command: str) -> ToolResult:
        try:
            import subprocess
            result = subprocess.run(
                command,
                shell=True,
                capture_output=True,
                text=True,
                timeout=60
            )
            output = result.stdout + result.stderr
            return ToolResult(
                status=ToolResultStatus.SUCCESS,
                output=output or "(无输出)"
            )
        except subprocess.TimeoutExpired:
            return ToolResult(
                status=ToolResultStatus.ERROR,
                error="命令执行超时"
            )
        except Exception as e:
            return ToolResult(
                status=ToolResultStatus.ERROR,
                error=str(e)
            )


# ============== 步骤 2: 系统提示词 ==============

CODE_AGENT_PROMPT = """你是一个 AI 编程助手，类似于 Claude Code。

你遵循 ReAct 模式（思考 - 行动 - 观察）来帮助用户完成编程任务。

## 可用工具

{tools_description}

## 工作流程

1. **理解任务**：分析用户的需求
2. **获取上下文**：读取相关文件了解当前代码
3. **分析修改**：确定需要做什么修改
4. **执行修改**：编辑文件或运行命令
5. **验证结果**：运行测试或命令确认修改成功

## 输出格式

- 思考时使用：Thought: [你的推理]
- 使用工具时使用：[调用工具]
- 完成任务时使用：Final Answer: [完整回答]

## 注意事项

- 编辑文件前先读取内容
- 运行命令前先说明将要执行什么
- 遇到错误时分析原因并尝试修复
- 修改完成后运行测试验证

请用中文思考和回答。"""


# ============== 步骤 3: 主函数 ==============

def main():
    """运行代码助手"""
    print("=" * 60)
    print("简易代码助手 - 基于 ReAct 模式")
    print("=" * 60)

    # 初始化组件
    llm = BailianClient(model="qwen-plus")

    tools = ToolRegistry()
    tools.register(ReadFileTool())
    tools.register(EditFileTool())
    tools.register(RunCommandTool())

    memory = ShortTermMemory(max_messages=50)

    config = AgentConfig(
        name="CodeAssistant",
        system_prompt=CODE_AGENT_PROMPT,
        max_iterations=15,
        verbose=True,
    )

    # 创建 Agent
    agent = ReActAgent(
        llm_client=llm,
        config=config,
        tools=tools,
        memory=memory,
    )

    # 交互式循环
    print("\n代码助手已就绪！输入 'quit' 退出。")
    print("示例：'帮我查看 src/main.py 并修复 bug'")
    print("-" * 60)

    while True:
        user_input = input("\n>>> ").strip()
        if user_input.lower() in ['quit', 'exit', 'q']:
            print("再见！")
            break
        if not user_input:
            continue

        try:
            result = agent.run(user_input)
            print(f"\n答案：{result.output}")
        except Exception as e:
            print(f"\n错误：{e}")


if __name__ == "__main__":
    main()
```
